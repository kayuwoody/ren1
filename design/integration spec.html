Architecture

Three independent systems communicate through a shared cloud layer. The cloud is the single source of truth; the POS and customer app are both clients.

Customer app → POSTs a new order to cloud API after payment is confirmed.
Cloud API → persists order, broadcasts a realtime event to subscribed POS clients.
POS app → receives event, displays new order on the Online Orders screen, drives KDS and label printer.
POS app → PATCHes order status back to cloud as barista progresses it.
Cloud API → sends customer notification when status = ready.
ℹ️
Why Supabase? You already use Vercel. Supabase gives you Postgres (structured orders + products), Row-Level Security, and a built-in Realtime channel — all on the free tier for your current volume. Your POS Node app subscribes via @supabase/supabase-js. No separate WebSocket server to manage.

Order flow

Happy path (online order)

  1. Customer selects items, customises, picks counter or curbside pickup.
  2. Customer proceeds to checkout → payment collected via PSP (Fiuu / iPay88).
  3. PSP webhook fires payment.success to your cloud API.
  4. Cloud API creates order record with status: "pending", broadcasts realtime event.
  5. POS receives event → plays alert tone → shows card in "New · Pending" column.
  6. Barista taps Accept & Print → POS PATCHes status to accepted → label prints + KDS updates.
  7. Barista makes drink → taps Mark Ready → POS PATCHes status to ready.
  8. Cloud API triggers customer notification (push / WhatsApp / SMS).
  9. Customer collects → barista taps Collected → status → collected. Sale recorded.
⚠️
Payment before order. Never create an order record until payment succeeds. Use your PSP's webhook + idempotency key to avoid duplicate orders on retry.

Called by the customer app after PSP payment confirmation. Idempotent on payment_ref.

PATCH /api/orders/:id

POS calls this to advance order status. Each status transition has server-side guards (e.g. can't move from pending to ready directly).

PATCH /api/orders/:id Advance order status 
// Accept order
{ "status": "accepted" }

// Mark ready (triggers customer notification)
{ "status": "ready" }

// Mark collected
{ "status": "collected" }

// Reject with reason
{ "status": "rejected", "reject_reason": "out_of_stock" }

// Pause intake (toggle)
{ "intake_paused": true, "outlet_id": "shell-seksyen-13" }

WebSocket / Realtime events

The POS subscribes to a Supabase Realtime channel on startup. Events flow cloud → POS.

✓
In Supabase, enable Realtime on the orders table. Filter by outlet_id = 'shell-seksyen-13' so multi-outlet expansion works for free later.

POS subscription (Node.js)

const { createClient } = require('@supabase/supabase-js')
const supabase = createClient(SUPABASE_URL, SUPABASE_ANON_KEY)

supabase
  .channel('orders-queue')
  .on('postgres_changes', {
    event:  '*',
    schema: 'public',
    table:  'orders',
    filter: `outlet_id=eq.shell-seksyen-13`
  }, (payload) => {
    const order = payload.new
    if (payload.eventType === 'INSERT') handleNewOrder(order)
    if (payload.eventType === 'UPDATE') handleOrderUpdate(order)
  })
  .subscribe()

The POS sends these on user action (status changes, stock updates, intake pause).

// When barista accepts an order
await supabase

// When barista marks ready

Label printer (Bluetooth)

The POS already controls the Bluetooth label printer. Wire it into the handleNewOrder callback after barista accepts.

function printLabel(order) {
  const lines = [
    `COFFEE OASIS #${order.id}`,
    `${order.customer.name} · ${order.pickup_type.toUpperCase()}`,
    `${formatTime(order.created_at)}`,
    '---',
    ...order.items.map(i =>
      `${i.qty}x ${i.product_name}${i.mods?.size === 'm' ? ' [L]' : ''}`
    ),
    i.mods?.notes ? `NOTE: ${i.mods.notes}` : '',
    '---',
    `TOTAL: RM ${order.total_paid.toFixed(2)} PAID`,
  ]
  printer.print(lines.join('\n'))   // use your existing printer driver
}
ℹ️
Print labels on Accept, not on order creation. This prevents printing for orders that get rejected. Add a "Reprint" button that re-calls printLabel(order) with the existing order object.

Kitchen Display System

The KDS is a separate screen driven by the POS. On Accept, open a WebSocket or push directly to the KDS frontend.

// Simple approach: KDS page subscribes to same Supabase channel
// filtered to status = 'accepted' | 'ready', sorted by accepted_at ASC
// No separate server needed.

const kdsOrders = await supabase
  .from('orders')
  .select('*')
  .eq('outlet_id', 'shell-seksyen-13')
  .in('status', ['accepted', 'ready'])
  .order('accepted_at', { ascending: true })

Customer notifications

Triggered server-side when order status becomes ready. Recommended: WhatsApp via Twilio or 360Dialog (both have Malaysian carrier support and good deliverability vs SMS).

// Supabase database webhook → Vercel edge function
// fires when orders.status changes to 'ready'

export default async function handler(req) {
  const { record } = await req.json()
  if (record.status !== 'ready') return Response.json({ ok: true })

  await twilio.messages.create({
    from:  'whatsapp:+60XXXXXXXXX',
    to:    `whatsapp:${record.customer_phone}`,
    body:  `☕ Your Coffee Oasis order #${record.id} is ready! ` +
           `Head to the counter at Shell Seksyen 13.`
  })
  return Response.json({ ok: true })
}

⚠️
Don't put your service_role key in the customer app. The customer app should only use the anon key + RLS policies. The POS (trusted, local machine) can use the service key for admin writes.

Stock sync rules

  • Drinks — no stock count. Mark unavailable manually (POS toggle) when you run out of a key ingredient. Auto-resume next day.
  • Pastries / packaged items — integer stock count in DB. Decrement on order.accepted. At count = 0 → available: false auto-set by Supabase trigger.
  • Threshold — set available: false client-side at count ≤ 2 as a buffer for walk-in conflicts. Server enforces at 0.
  • Conflict handling — if an online order slips through for a just-sold-out item, the barista rejects it with reason out_of_stock → auto-refund fires via PSP API → customer gets WhatsApp notification.
  • POS sync frequency — POS pushes stock counts to cloud every 60 seconds passively, and immediately on any manual stock edit.

Auth & security

  • Customer app: Supabase anon key + RLS. Customers can INSERT orders; can only SELECT their own orders (by session token). Cannot read other customers' orders.
  • POS: service_role key stored as environment variable on Windows machine. Rotate quarterly.
  • PSP webhook: verify signature header before processing. Reject anything without a valid HMAC.
  • Rate limit POST /api/orders to 5 req/min per IP at the Vercel edge to prevent abuse.
  • Idempotency: payment_ref must be unique per order. DB unique constraint + 409 on duplicate.
